Radius 医疗数据管理平台 - 完整微服务架构设计与落地
技术负责人 / 架构设计 | 2025.03 - 至今 | 生产系统
项目定位：对原有 Python 单体系统进行整体重架构，构建公司新一代医疗数字化基础设施，承载患者信息管理、医学影像（DICOM）处理、癌症数据分析、数据湖、权限管理等全部业务域。作为独立架构师完成从技术选型、架构设计到全量代码交付的全链路工作。
技术架构：Go 1.24、CloudWeGo（Kitex RPC + Hertz HTTP）、PostgreSQL、etcd、Iceberg-REST、Polars、PyArrow、Casbin、Docker、Google Wire、Thrift IDL
系统规模：10个业务微服务 + API 网关，按业务域独立部署；48个 Thrift IDL 定义服务契约，IDL-First 驱动开发
服务列表：identity、datalake、patient、cancer、dicom、medrec、ocr、athoslide、permission + API网关
架构决策与核心实现：
  1. 服务边界设计：以业务能力（而非数据层）为准则划分 10 个微服务，规避跨服务分布式事务，保障各服务独立演进；DDD 四层架构（Handler→Logic→DAL→Converter）+ 仓储模式，将业务逻辑与框架彻底解耦，服务可独立替换底层存储
  2. 医疗数据湖：设计 datalake_srv 服务，基于 Iceberg-REST 协议管理医疗数据元数据，结合 Polars + PyArrow 实现高性能数据处理，支持跨数据源联合查询，为数据分析层提供统一数据访问接口
  3. 多级权限体系：基于 Casbin 实现 RBAC 权限模型，设计组织→部门→用户三级权限结构，实现菜单权限动态解析和 API 级鉴权，满足多院区、多科室的差异化权限管理需求
  4. 可观测性体系：设计 RPC 追踪中间件（request_id/trace_id 全链路透传），建立统一 6 位数字错误码体系，实现分层错误转换，接入 OpenTelemetry 链路追踪与监控告警
  5. 工程化体系：IDL-First 开发模式（48个 Thrift IDL 先于实现），Google Wire 编译时依赖注入，完整 Docker 编排方案，环境驱动配置管理，2,000+ 行架构文档支撑团队协作

典型线上事故处理：
  • 服务器死机恢复：Redis 升级时 gcc 源码编译污染内核，服务器重启失败——确认备份完整后指导现场重装系统，在公司同规格服务器上交叉编译 Redis 二进制包传送，完成零依赖升级，系统当日恢复
  • SSH 失效依赖修复：安全补丁安装引发 apt 依赖链断裂，SSH 不可用——通过 nc 逐一传输缺失 deb 包，在无 SSH 环境下完成全部服务依赖恢复
  • 语音合成卡顿根治：Python ctypes 内嵌调用 C 模块引发系统级卡顿——主动介入，将 C 代码重构为接收命令行参数的独立可执行程序，改用 subprocess 进程隔离调用，彻底解决卡顿
  • 其他：IP 变更后 HTTPS 证书失效（内网自建 CA），用旧 CA 重签新 IP 证书快速恢复；MongoDB 数据误重置，通过操作执行记录逐条还原
